Github 명령어(Command)

⭐️ Github 업로드와 명령어

원격저장소(Remote Repository)인 Github에 저장소(Repository)를 생성하였다면, 이 저장소(Repository)에 업로드하여 저장시키고, 각종 명령어들도 알아보자.

⭐️ Github 명령어

Github의 저장소(Repository)에 업로드를 하기 위해 Github에서 사용되는 간단한 명령어들(Commands)를 알아야 사용이 가능하다. 명령어들을 알아보고 사용해보자. 모든 명령어는 cmd / 터미널(Terminal) / git bash / VS code 의 터미널 중에서 편한것을 선택하여 작성하면 된다. 작성자는 VS code의 터미널을 주로 사용하고 있다.

  • git init : 현재 지정된 폴더에 .git 이라는 빈 파일을 생성시켜 git으로 관리 할 수 있도록 만들어준다. [초기화 / 맨 처음 프로젝트를 올릴 때 사용]

  • git status : 현재 폴더안의 파일들의 상태를 보여준다.

    • Changes to be commited : 스테이지에 추가된 파일 리스트 [git add 명령어 사용한 상태]

    • Changes not staged for commit : 스테이지에서 추적이 되었던 파일이 수정되었지만, 스테이지로 추가가 되지 않은 상태 [git add 명령어로 스테이지에 추가가 되었지만 그 후에 수정이 일어난 상태]

    • Untracked files : 신규로 생성된 파일 리스트

  • git add : 현재 작업중인 파일을 버전관리를 시작하겠다고 선언하는 부분이다. [스테이지로 추가]

    • git add 파일명 : 특정 파일만 버전관리를 시작하겠다고 선언하는 것이다. [특정 파일만 스테이지로 추가]

    • git add . : [.]이 모두라는 뜻으로 통한다. 특정 파일이 아닌 현재 폴더 안의 모든 파일의 버전관리를 시작하겠다고 선언하는 것이다. [모든 파일을 스테이지로 추가]

  • git commit -m "설명" : git add로 버전관리를 시작하겠다고 선언한 내용들을 버전으로 만들어서 관리한다. 뒤에 설명란에는 수정한 내용이나 추가작업한 내용을 적어주면 추후에 알아보기가 쉬우며, 원하는 지점으로 돌아갈 때, 식별하기 편리하다. [스테이지에 추가된 내용을 버전으로 생성]

  • git remote add origin "원격저장소 주소" : 버전을 생성한 후, 원격저장소에 업로드를 하기위해 원격저장소를 연결한다.

  • git remote -v : 현재 폴더와 연결되어있는 원격저장소를 나타낸다.

  • git push origin master : git commit 으로 버전을 생성하고, 원격저장소와 연결이 되었다면 업로드를 하기 위해 push를 사용한다. [생성된 버전을 master 브랜치로 저장]

  • git log : git commit 으로 생성한 버전(히스토리)을 최근순으로 조회한다. git log의 속성도 여러가지가 존재한다.

    • git log -p : 각 commit의 diff 결과를 보여준다.
      diff : commit의 결과를 비교하여 변경된 곳을 자세히 알려준다.

    • git log -(숫자) : (숫자)만큼의 commit을 최신순으로 보여준다. ex) -2 를 입력하면 최신순으로 2개의 commit이 나타난다.

    • git log --stat : 각 commit의 파일별 변경사항 등 통계를 볼 수 있다.

    • git log --pretty : commit의 히스토리를 보여줄 때 기본형식 이외에 사용자가 직접 정의한 내용으로 보이게 지정할 수 있다.

      • git log --pretty=oneline : --pretty 속성에 =oneline 을 같이 작성하여 사용하면, 각 commit을 한줄로 표현하여준다. 이 때, =oneline 자리에 short, full, fuller 옵션을 넣을 수 있는데 이 속성은 정보를 조금씩 가감해서 보여준다.

      • git log --pretty=format : --pretty 속성에 =format 을 사용하면 사용자가 원하는 포맷으로 결과를 출력 할 수 있다.
        ex) git log --pretty=format:"%h -%an, %ar : %s"


        format의 옵션
        %H 커밋 해시 %h 짧은 길이 트리 해시
        %T 트리 해시 %t 부모 해시
        %P 부모 해시 %p 짧은 길이 부모 해시
        %an 저자 이름 %ae 저자 메일
        %ad 저자 시각 %ar 저자 상대적 시각
        %cn 커미터 이름 %ce 커미터 메일
        %cd 커미터 시각 %cr 커미터 상대적 시각
        %s 요약

    • git log --graph : commit의 branch와 merge의 히스토리 정보까지 ASCII 그래프로 나타낸다.

    • git log --branches : branch별 최신순으로 commit한 내용이 히스토리를 보여준다.

    • git log 속성=값 : commit을 조회하는 조건을 등록하여 조건에 알맞는 내용의 commit의 히스토리만 불러온다.
      ex) git log --since=2.weeks [지난 2주 동안 만들어진 커밋 조회]


      조건을 걸어주는 속성
      --since, --after 명시한 날짜 이후의 커밋만 검색
      --untill, --before 명시한 날짜 이전의 커밋만 조회
      --author 입력한 저자의 커밋만 검색
      --committer 입력한 커미터의 커밋만 검색
      --grep 커밋 메시지 안의 텍스트를 검색
      -S 커밋 변경(추가/삭제)된 내용 안의 텍스트 검색

    • git reset HEAD^ : 가장 최신의 commit을 취소하고 바로 이전의 commit으로 돌아간다

    • git reset HEAD~[취소할 commit의 수] : 가장 최신의 commit을 [취소할 commit의 수]만큼 취소하고 돌아간다.

    • git reset [commit의 6자리숫자] --soft : 지정된 [commit의 6자리숫자] 히스토리로 이동하지만, 작업중인 파일과 git add로 올린 스테이지의 내용은 이동하지 않고 남아있다.
      ex) 5번째 Commit에서 3번째 Commit으로 이동 시 Commit은 3번째로 돌아가지만, 작업중인 파일과 스테이지의 내용은 Commit은 5번째인 그대로 남아있는다. [git commit 으로 바로 다시 버전 생성가능]


    • git reset [commit의 6자리숫자] --mixed : 지정된 [commit의 6자리숫자] 히스토리로 이동하며 git add로 올린 스테이지의 내용도 같이 이동하여 사라지만, 작업중인 파일 내용은 이동하지 않고 남아있다.
      ex) 5번째 Commit에서 3번째 Commit으로 이동 시 Commit과 스테이지의 내용은 3번째로 돌아가지만 작업중인 파일은 그대로 남아있다. [git add를 진행하여 스테이지에 등록 후 git commit을 진행하여야 한다.]


    • git reset [commit의 6자리숫자] --hard : 원하는 commit의 히스토리로 이동하며, 그 이후에 작성된 commit은 전부 삭제된다.
      ex) 5번째 Commit에서 3번째 Commit으로 이동 시 Commit과 스테이지의 내용, 작업중인 파일의 내용도 모두 삭제되며 돌아간다.
